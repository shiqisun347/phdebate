## 0. 全局新增能力：辩手形象

当前 `Speaker` 数据无任何形象字段，而「阵容介绍」「比赛实况发言动效」均依赖形象展示。小七已有 `image_url`，本次为辩手补齐对齐能力。

- **数据模型**：为辩手（agent 与人类）增加形象图字段（建议 `image_url: string`，与小七一致）。
- **管理页字段**：在「Agent 管理」与「辩手管理」中新增「形象图」上传/填写字段，供后期由控场人员上传真实形象。
- **默认形象**：先为每位 agent 生成一张默认形象图（统一风格的占位/示意形象），使大屏动效在未上传真实图时也能正常呈现。
- **回退策略**：无形象图时——AI 辩手使用默认 AI 形象；人类辩手使用「姓名首字 + 阵营色」的圆形文字头像（沿用现有规则）。

## 1. 大屏场景（`/screen`）

大屏共 9 个场景，本次改动范围如下表，未列「保持现状」的场景不改：

| # | 场景 | 现状映射 | 本次处理 |

| 2 | 阵容介绍 | 当前并入 idle | **重新设计**（独立场景 + 形象动效） |
| 3 | 比赛实况 | `live` | 改进（中间发言动效 + 字幕条降高） |

### 1.1 候场页面
- 去掉「请等待主持人宣布开始」一句。
- 其余信息（赛事抬头、辩题、左右阵容）保持不变。

### 1.2 阵容介绍（重新设计）
- 作为独立场景，由控场台「阵容介绍」按钮触发。
- 主持人介绍某位 agent 后，agent 进行**自我介绍**；大屏需在此场景中央**展示当前介绍中 agent 的形象**。
- 形象为**静态图片**，但叠加**动态动效**营造「正在说话」的感觉：
  - **声波/律动动效**（围绕形象的声波环、底部声波条等）。
  - **AI 形象的动效更具辨识度、更灵动**（如更丰富的呼吸光晕、粒子/声波、阵营色渐变），人类形象动效相对克制。
- 实现可参考开源「说话头像 / 声波可视化」类项目的视觉语言（如 voice-wave、siri-wave 风格的 CSS/Canvas 声波）。
- 左右两侧仍保留阵容名单，高亮当前正在自我介绍的辩手。

### 1.3 比赛实况（改进）
- **中央新增发言动效区**：当人类或 AI 辩手发言时，中央展示类似「阵容介绍」的形象 + 声波动效，但**尺寸略小**，给环节信息/计时让出空间。
  - **人类与 AI 均展示**；**AI 形象动效更具辨识度、更灵动**，人类相对简洁。
- **字幕条降高**：当前底部发言窗口过高，改为**单行文本**高度（仍保持固定大小、流式滚动到底部的现有行为，仅压缩高度为一行）。
- 其余（左右阵容高亮、环节名、计时、自由辩论多计时、AI 思考态、等待主持确认态）保持现有逻辑。

### 1.4 小七点评（重新设计）
- 展示**小七形象**（静态图 + 声波/律动动效，规则同 1.2，使人感觉小七正在说话）。
- **不需要发言窗口/字幕**。
- 触发自控场台「小七控制」。

### 1.5 小七评判 / 小七结果（参考评委结果页改进）
- 视觉结构参考现有「评委结果」场景。
- 展示数据：**获胜方 + 最佳辩手**（**不展示理由**）。
- 数据来源：由控场人员在「小七结果录入模块」**手工录入**获胜方与最佳辩手（小七不自动产出理由）。

### 1.6 致谢环节（新增）
- 以**一句致谢语**为主体，居中大字呈现（配赛事抬头/logo 即可）。
- 致谢语文案可后续配置；先以默认文案占位（如「感谢各位的参与」）。

### 1.7 不改动场景
- 评委点评、评委结果、学生结果：保持现状。

## 2. 控场台页面（`/admin → 控场台`）重构

整体目标：**让控场人员只面对与当前阶段相关的内容**，按真实交互场景优化效率、体验与误操作风险。页面自上而下重构为 5 段（未提及的历史模块全部移除）。

### 2.1 顶部信息条（压缩）
- **去掉**「现场流程编排与控制」标题/描述。
- 显示「当前比赛：2026 年"学有所承"之人机大战」，**不换行**。
- 压缩该区域高度，仅作为轻量标题条。

### 2.2 比赛实况状态栏（原「控制台模块」位置改为此）
- 作为一个**不高的状态栏**，显示当前比赛实况：
  - 当前环节、当前发言人、发言状态、倒计时等。
- 不占用过多纵向空间。

### 2.3 流程控制模块（原控制台模块）
- **去掉顶部那行**「进行中　大屏：比赛实况　2026 年"学有所承"之人机大战」（信息已在 2.1/2.2 体现）。
- **按钮重新布局与样式**：
  - 「开始比赛」与「重置比赛」归为一组，**统一放到最右侧**。
  - **「下一步骤」按钮**为核心：
    - 使用**最高亮度的强调色**，放在**最显眼、最易操作**的位置。
    - 按钮文案体现**下一步具体是什么**（如「发布小七结果」「进入下一环节」「等待评委合议」等动态文案）。
    - **需要控场人员操作时**：按钮可点击，并带**闪烁/呼吸动画**提示。
    - **不需要操作时**：按钮为**灰色、不可点击**状态。
  - **去掉「紧急停止」按钮**；「重置比赛」使用与原紧急停止相同的（警示）配色。
- **功能修复**：
  1. 「开始比赛」**仅在最开始阶段可选**；比赛开始后不可再点。
  2. 「暂停 / 继续」合并为**一个切换按钮**：暂停后变「继续」，继续后变「暂停」。
  3. 所有确认改为**本 UI 的确认弹窗**（替换浏览器原生 `confirm`）。
  4. 「重置比赛」**只需点击确认**即可，**不再要求输入文字**确认。

> 说明：「下一步骤」的文案与可点击/闪烁状态，依据后端流程状态（如 `flow.awaiting_host_confirm`、`flow.next_action`、当前场景与环节）动态推导。

### 2.4 阶段导航条（页内导航，新增）
- 一条**不高的导航条**，含 4 个选项：**赛前阶段 / 比赛过程 / 小七点评 / 评委与学生点评**。
- 作用：切换下方内容区显示，相当于页内 tab。
- **根据当前比赛阶段自动切换**到对应选项；**也允许用户手动切换**。
- 建议的阶段 → 选项自动映射（可在实现时微调）：

  | 比赛阶段 / 场景 | 自动定位选项 |
  |---|---|
  | 草稿 / 待开始、辩题介绍、阵容介绍 | 赛前阶段 |
  | 比赛进行中（比赛实况） | 比赛过程 |
  | 小七点评 / 小七评判 | 小七点评 |
  | 评委点评 / 评委结果 / 学生结果 / 致谢 | 评委与学生点评 |

### 2.5 阶段内容区（随导航条切换）
各选项对应的下方模块如下（后续可能补充更多模块）：

- **赛前阶段 / 比赛过程**：当前仅含 **AI 辩手控制模块**。
  - 当前 AI 辩手控制**排版不合理、不美观、占空间过大**，需**完全重新设计**（紧凑卡片/行式布局，突出「当前发言/可操作」状态与关键动作）。
- **小七点评**：含 **小七控制模块** + **小七结果录入模块**。
  - 按新空间**重新设计小七控制模块**，贴合控场人员使用习惯，设计合理交互、提升效率与体验。
- **评委与学生点评**：含 **评委结果录入模块** + **学生结果实时统计模块** + **学生投票管理模块**。
  - 学生投票管理模块当前仅含「开启学生投票 / 关闭学生投票」两个功能。

### 2.6 模块清理
- **未提及的历史模块一律移除**（如旧的「大屏场景切换 10 步流程」「合并在控场台底部的实时辩论过程折叠块」等不再保留，实时辩论过程改为独立页见 §3）。

## 3. 实时辩论过程（独立页面）

- 在「现场控制」分区下，**独立成一个页面**。
- 页面包含：**控制区** + **类似聊天框的辩论历史区**。
- 用途：比赛过程中，控场人员对历史辩论进行**查看、修正**等操作。
- **可修改历史辩论内容**：
  - 修改后需弹出**确认窗口**，确认后才生效。
  - 修改后的内容与**全局维护的历史辩论内容同步**。
  - 后续发给 agent 的请求，将**基于修改后的历史辩论内容**进行（即 `debate_history` 取已修正版本）。

## 4. 其他需求

### 4.1 日志：从「请求日志」改为「日志查看」
- 名称与定位：由「请求日志」改为「**日志查看**」。
- 内容：应包含**完整的日志输入与输出**内容（而非仅请求元数据）。
- **统计粒度细化**：当前粒度过粗、且很多事件未统计；需补全应记录的事件，并按**多级分类**组织，便于筛选与定位。

#### 多级分类（筛选维度）
日志按以下层级分类，每一层都可作为「日志查看」页的筛选项（可叠加）：

- **第 1 级 · 性质**：区分这条日志是真实比赛还是调试。
  - `正式`：比赛运行过程中真实产生的请求/事件。
  - `测试`：从后台调试工具触发的（Agent 调试、ASR/TTS 流式测试、API 自检等）。
- **第 2 级 · 类型**：
  - `agent`：AI 辩手请求（首次请求 / 重试 / 中断 / 替代代答）。
  - `语音`：ASR、TTS（含流式分段）。
  - `小七`：自我介绍 / 点评 / 结果 / 自定义问题命令。
  - `审计`：流程推进、场景切换、结果录入与公布、学生投票开关、历史修正等控场操作。
- **第 3 级 · 时机（上下文）**：标注该日志发生时的现场上下文，用于按阶段/流程回溯。
  - `比赛阶段`：发生时所处环节（如「正方一辩立论」「自由辩论」）。
  - `流程节点 / 场景`：发生时所处流程或大屏场景（如「阵容介绍」「小七点评」「评委结果」）。

#### 每条日志的字段（输入 + 输出全量）
- 时间、性质、类型、时机（阶段 + 场景）、操作者/发起端。
- 状态（成功 / 失败）、延迟。
- **完整输入**（请求体 / 参数 / 触发上下文）。
- **完整输出**（返回内容 / 转写文本 / 合成结果引用 / 错误详情）。

> 实现时以上述三级为筛选骨架；具体「类型」下还需记录的细分事件清单，可在实现阶段对照后端各调用点补全（见 §6）。

### 4.2 导航页（`/`）
- **不再包含「主持导播台(host)」**入口。

### 4.3 辩手端（`/console`）
- 身份选择中**仍可完整展示**所有席位（含 agent 席位，作只读展示）。
- 但 **agent 席位不可被选择**（人类选手只能选人类席位进入；agent 席位置灰/不可点击）。

## 5. 已确认的决策（来自负责人）

1. **辩手形象来源**：为 agent 生成默认形象图，同时在管理页新增形象字段，供后期上传真实图。
2. **比赛实况发言动效范围**：人类与 AI 均显示；**AI 形象更具辨识度、更灵动**。
3. **致谢环节内容**：以**一句致谢语**为主。
4. **小七结果展示数据**：**获胜方 + 最佳辩手**，**不要理由**，由控场人员手工录入。
5. **日志分类**：采用「性质（正式/测试）→ 类型 → 时机（阶段/流程）」三级分类（见 §4.1）。
6. **辩手端**：完整展示所有席位，但 **agent 席位不可选**（见 §4.3）。
