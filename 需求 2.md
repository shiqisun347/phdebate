

host 开始比赛->主屏幕切换到实况界面，按照预设的流程进行比赛，自动控制发言时间和发言人切换；host 可以随时暂停和继续比赛，重置当前发言或者重置整个比赛；比赛结束后，host 手动切换到评委点评界面，学生开始投票；host 手动切换到评委结果界面，宣布获胜方，同时自动结束学生投票；host 手动切换到学生投票结果界面，展示学生的投票结果和最终的获胜方。

按照预设的流程进行发言，倒计时自动开始，但对于人类辩手，发言的开始、暂停、继续和结束由人工控制；对于 AI 辩手，发言的开始和结束由后台自动控制，host 和后台可以控制、重置 ai 的发言。每一个阶段的发言时间到了，主持人需要手动点击下一阶段，自动切换发言人。比赛需要维持一个全局唯一的辩论过程，重置时，admin 端应该可以像聊天记录一样，看到每个阶段的发言内容和时间记录，方便管理员进行干预和调整。

自由辩论阶段，人工辩手页面会有一个跳过按钮，如果两个人都点了跳过或超过5s，那么则agent回答；自由辩论是交替的，单次发言不超过30s；自由辩论时，同一方的 agent 是纯随机选择，也意味着可以连续发言；

人工辩手页面，应该要展示下一个发言的选手是谁，开始发言后，可暂停继续，但是如果选结束发言以后，不能再继续，倒计时直接结束；


TTS 和 ASR的管理在 admin 页面，应该可以添加 api 之类的设置；TTS 和 ASR的时机和流程是固定的，tts 的参数由 api 决定，需要提供一个 md文件，协商好交互的输入输出。参考常见的、主流的交互方式。

如果下一个是需要 agent 辩手发言，应该流式调用 ASR，然后把 ASR 的结果实时展示在主屏幕上；ASR 结束后，后台调用 agent 接口，流式获取生成的文本内容，同时流式调用 TTS 接口，把生成的语音播放出来，同时把文本内容展示在主屏幕上；


agent的调用时，使用 restful 方式，提供接口地址、请求方式、请求参数等信息，具体的，需要提供结构化的辩论文本过程，这个最重要，其次，是身份比如x 方 x 辩、agent_name，阶段，比如自由辩论等
