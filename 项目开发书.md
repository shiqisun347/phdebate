# 人机辩论赛控制系统项目开发书

版本：v1.0  
日期：2026-06-12  
基线：`需求.md` 与当前项目代码  
读者：项目开发团队、测试人员、现场联调人员

## 0. 文档定位

本文是针对当前 `phdebate` 项目的改进开发书，目标是把已有原型系统推进为可用于单场线下人机辩论赛的正式现场系统。

本文是独立文档，不要求读者阅读其他开发文档。本轮改造以 `需求.md` 和当前代码实现为事实基础，以本文为执行依据。

本文不直接包含实现代码，不替代后续 PR；它用于明确开发目标、模块边界、接口方向、数据保存要求、阶段计划和验收标准。

优先级约定：

- `P0`：必须先做，否则后续改造会反复返工。
- `P1`：核心现场能力，必须在正式演示前完成。
- `P2`：增强体验、可维护性或复盘能力，建议在完整彩排前完成。
- `可延后`：保留方向，不纳入本轮必要交付。

## 1. 项目现状理解

### 1.1 当前前端入口

当前前端入口由 `apps/frontend/src/App.tsx` 直接根据 `window.location.pathname` 分发：

| 路径 | 当前页面 | 当前问题 |
| --- | --- | --- |
| `/` | `NavigationPage` | 暴露比赛 ID 输入；学生投票入口与现场主入口混在一起 |
| `/host` | `HostPage` | 现场主持功能、导播功能、投票公布、屏幕切换混在一起，现场优先级不清 |
| `/admin` | `AdminPage` | 已有模块化雏形，但比赛管理、辩手管理、Agent 管理仍未独立成清晰业务对象 |
| `/screen` | `ScreenPage` | 仍保留 `teams/intermission/result` 等旧场景语义，和新需求的评委点评/评委结果/学生结果不一致 |
| `/console`、`/console/:speakerId` | `ConsolePage` | 已有身份选择和硬件测试，但人类/AI 辩手页面仍在同一组件内，发言暂停/继续缺失 |
| `/vote`、`/vote/:matchId` | `VotePage` | 仍以 `matchId` 为入口心智；Logo 不符合新需求；防重复较弱 |

当前 `App.tsx` 通过 `match_id` 查询参数、`/vote/:matchId` 或默认 `match_001` 获取比赛 ID。这与新需求“当前项目只能进行一个辩论，普通链接不需要带比赛 ID”冲突。

### 1.2 当前后端入口

当前后端主要在 `apps/backend/app/main.py` 中定义 REST 和 WebSocket：

- `GET /api/matches/{match_id}`：读取快照。
- `POST /api/matches/{match_id}/start|pause|resume|finish`：比赛控制。
- `POST /api/matches/{match_id}/phases/next`：进入下一环节。
- `POST /api/matches/{match_id}/screen/scene`：切换大屏场景。
- `POST /api/matches/{match_id}/speakers/{speaker_id}/start-speaking|stop-speaking`：发言控制。
- `POST /api/matches/{match_id}/votes` 与 `/audience-votes/*`：投票与公布。
- `POST /api/demo/reset`：重置 demo 状态。
- `GET /screen`、`/host`、`/admin`、`/console`、`/vote`：托管前端 SPA。
- `GET /vote/{match_id}`、`GET /console/{speaker_id}`：兼容带参数页面。
- `WebSocket /ws/matches/{match_id}`：实时状态通道。

当前所有核心控制仍以 `match_id` 路径参数为主，没有“当前比赛 current”抽象。旧比赛归档与新比赛生成也没有正式机制。

### 1.3 当前状态与存储

当前主要状态源是 `apps/backend/app/services/match_store.py` 中的 `MatchStore`：

- 内存中维护完整 `snapshot`。
- `reset_demo()` 固定生成 `match_001`。
- `events` 保存最近事件，并通过 WebSocket 广播。
- `_refresh_clocks()` 只在读取快照或部分操作时刷新时钟，不负责主动推进流程。
- 发言、投票、转写、音频等核心数据主要仍存在快照中。

当前 SQLite 层是 `apps/backend/app/services/sqlite_repo.py`：

- `app_state` 保存快照 JSON。
- `events` 保存事件日志。
- `audit_logs` 保存主持/管理员操作审计。

这已经具备“可恢复”的基础，但还不满足“所有辩论相关数据结构化保存、方便查询分析”的新需求。

### 1.4 当前主要问题

1. `match_id` 心智外露  
   前端链接、API、WebSocket、投票页、导航页仍处处暴露 `match_id`。正式现场只需要“当前比赛”。

2. Host 现场优先级混乱  
   Host 应该服务主持人管理流程与时间，不应该承担复杂后台管理、危险操作和低频调试。

3. 大屏场景和新流程不匹配  
   新需求需要候场、实况、暂停、评委点评、评委结果、学生投票结果。当前仍是 `idle/opening/teams/live/intermission/result`。

4. 结果页表达不足  
   当前结果页把评委和学生结果放在同一 `result` 语义里，缺少官方评委结果和学生投票结果的分离，也缺少仪式感。

5. 发言流程不完整  
   人类辩手可以开始/结束发言，但缺少发言暂停/继续；发言重置也未形成正式流程。

6. 自动计时推进不足  
   当前时钟会到期变为 `expired`，但没有后台 timer loop 主动发事件、结束发言、提示主持人确认下一步。

7. 投票防重复偏弱  
   当前前端生成 localStorage token，后端记录 `used_audience_tokens`，但缺少 IP/UA hash 级无感辅助限制。

8. 数据结构化不足  
   快照和事件可以导出，但请求/响应、发言、投票、Agent 调用、音频、修订等还没有足够清晰的结构化表。

## 2. 改造目标

### 2.1 产品目标

本轮改造后，系统应支持一场线下 4v4 人机辩论赛从候场、开始、各阶段发言、暂停继续、评委点评、学生投票、评委结果、学生结果到归档导出的完整闭环。

核心目标：

- 单场当前比赛，不要求现场人员理解或输入比赛 ID。
- 主持人负责现场流程和时间控制。
- 技术后台负责配置、监控、Agent、语音、数据、应急。
- 大屏具备清晰状态、投影可读性和足够仪式感。
- 辩手端只告诉辩手“现在是什么阶段、是否轮到你、该做什么”。
- 所有关键比赛数据完整保存、可导出、可复盘。

### 2.2 非目标

本轮不做：

- 多场赛事排程系统。
- 历史比赛浏览界面。
- 完整评委独立投票页面。
- 学生实名登录或学号校验。
- 任意人数、任意赛制编辑器。
- 多租户权限系统。

## 3. 总体改造方案

### 3.1 单场当前比赛

目标状态：

- 所有正式页面默认访问当前比赛。
- 普通链接为 `/host`、`/admin`、`/screen`、`/console`、`/vote`。
- API 和 WebSocket 支持 `current` 作为当前比赛别名。
- 老链接保留兼容：`?match_id=match_001`、`/vote/:matchId`、`/console/:speakerId` 不立即删除。

改造任务：

- `P0`：前端 `App.tsx` 默认使用 `current` 或后端当前比赛摘要，不再在导航页暴露比赛 ID。
- `P0`：后端 `_ensure_match()` 支持 `current`。
- `P0`：`api/client.ts` 增加当前比赛路径封装，减少页面拼接 `matchId`。
- `P1`：清理页面文案和链接，不再展示 `match_001`。
- `P2`：导出包和音频路径仍使用真实 `match.id`，保证历史隔离。

验收标准：

- 浏览器直接访问 `/host`、`/admin`、`/screen`、`/console`、`/vote` 均可用。
- 导航页没有比赛 ID 输入框。
- 旧链接仍不报错。

### 3.2 半自动流程推进

目标状态：

- 固定环节发言人、发言时间由后台赛制配置决定。
- 到时后系统自动结束当前发言并提示 Host 确认下一步。
- Host 点击确认后进入下一环节或赛后流程。
- 不采用完全无人值守自动跳转，避免现场误切。

改造任务：

- `P1`：后端新增 timer loop，定期检查 running clock。
- `P1`：时钟到 0 时广播 `clock.expired`。
- `P1`：如果有当前发言，自动结束发言并广播 `speech.timeout`。
- `P1`：比赛状态增加“等待主持确认下一步”的可表达字段，例如 `flow.awaiting_host_confirm = true`。
- `P1`：Host 显示明确的下一步按钮和原因。

验收标准：

- 不操作页面时，倒计时到 0 后大屏和 Host 同步显示到时。
- 当前发言结束，发言按钮被锁定。
- 系统不会自动跳到错误环节，必须 Host 确认。

## 4. 页面改造规格

### 4.1 导航页

现状问题：

- `NavigationPage` 仍展示比赛 ID 输入。
- 学生投票入口和正式控制入口混在一起。
- 辩手端展示多个具体 speaker 链接，而新辩手页已经有身份选择流程。

目标状态：

- 导航页是现场工作人员入口页。
- 主入口统一展示：主持导播台、技术后台、大屏、辩手端。
- 不展示学生投票入口。
- 不展示比赛 ID 输入。
- 辩手端只保留一个 `/console` 入口。

改造任务：

- `P0`：删除比赛 ID 输入框。
- `P0`：删除学生投票卡片。
- `P0`：删除 4 个辩手快捷卡片，改为一个“辩手端”卡片。
- `P1`：导航页显示当前比赛标题、辩题和服务地址。

验收标准：

- 首页首屏只有四个工作入口。
- 点击辩手端进入身份选择页。

### 4.2 Host 主持导播台

现状问题：

- `HostPage` 里同时有流程列表、发言控制、计时、大屏场景、投票公布、结果页等功能。
- 仍提供阵容页、中场页、结果页等旧场景按钮。
- 主持人可以做过多不该在现场临时处理的操作。

目标状态：

Host 是现场主持流程台，只回答三件事：

1. 现在比赛处于什么状态。
2. 当前/下一步该做什么。
3. 关键控制按钮在哪里。

Host 保留：

- 开始比赛。
- 暂停比赛。
- 继续比赛。
- 重置当前发言。
- 结束当前发言。
- 确认进入下一轮或下一环节。
- 宣布辩论结束。
- 切评委点评并开启学生投票。
- 切评委结果并关闭学生投票。
- 切学生投票结果。
- 手动铃声。
- 设置现场唯一声音输出端为 Host 或 Admin，即“哪台控制电脑物理连接大屏/音响”。

Host 不负责在发言控制区选择本轮发言人。系统必须根据赛制规则自动计算当前可发言席位，并把开始发言权限授予对应辩手端；Host 只确认流程节奏、到时后的下一轮/下一环节，以及必要的结束/暂停/重置当前发言。

Host 不保留：

- 选择任意本轮发言人。
- 任意回滚。
- 任意跳过。
- 复杂时钟校准。
- Agent 中断/重试/代输入。
- ASR/TTS 诊断。
- 权限配置。
- 数据导出。

改造任务：

- `P1`：重构 Host 布局为“顶部比赛状态 + 中央当前流程 + 右侧下一步动作”。
- `P1`：删除旧大屏分段按钮中的“阵容/中场/结果”。
- `P1`：新增“当前发言重置”按钮。
- `P1`：新增赛后流程按钮组。
- `P1`：发言区改为“发言权限”只读状态，展示赛制自动授权的席位，不提供主持人手选发言人入口。
- `P1`：新增现场声音输出设置，Host/Admin 二选一作为唯一浏览器提示音输出端，可关闭；该设置表达的是哪台控制电脑连接大屏/音响，不要求辩手、投票或大屏页面外放。
- `P1`：按钮必须显示禁用原因。
- `P2`：给暂停、到时、等待确认增加清晰视觉状态。

验收标准：

- 1366x768 下 Host 首屏可见当前环节、当前发言、倒计时、暂停/继续、下一步。
- 暂停后 Host、Screen、Console 状态一致。
- Host 不出现后台配置和应急调试功能。
- 自由辩论和固定席位环节中，Host 只确认下一步，辩手端发言权限由规则自动下发。
- 手动铃声、时间提示和浏览器 TTS 只在当前指定的 Host 或 Admin 输出端电脑播放；辩手端、投票端、大屏页不播放。

### 4.3 大屏 Screen

现状问题：

- `ScreenPage` 仍保留 `TeamsScene`。
- `ScreenTop` 显示环节 x/x、WebSocket 状态等技术信息。
- `IntermissionScene` 语义不等于新需求中的评委点评。
- `ResultScene` 同时承载评委和学生结果，仪式感不足。

目标场景：

| 场景 | 说明 | 优先级 |
| --- | --- | --- |
| `idle` | 候场页 | `P1` |
| `live` | 比赛实况 | `P1` |
| `paused` | 比赛暂停覆盖态 | `P1` |
| `judge_commentary` | 评委点评 + 学生投票二维码 | `P1` |
| `judge_result` | 官方评委结果 | `P1` |
| `audience_result` | 学生投票结果 | `P1` |

候场页要求：

- 去掉最上面的大 logo。
- 队伍信息框中“正方 · 编程思维 / 反方 · 提问思维”更突出。
- 展示战队名、成员名、人类选手/AI 模型名。
- 框体紧凑，不做巨大卡片堆叠。

实况页要求：

- 右上角放计算所 logo。
- 左上角放“第一届人机辩论赛”赛事字标。
- 删除环节 x/x、open/connecting 等技术状态。
- 保留当前环节、当前发言、倒计时、双方信息和字幕。

评委点评页要求：

- 维持实况页背景与两侧队伍信息。
- 中央大字“评委点评”。
- 下方展示学生投票二维码和投票状态。
- 去掉底部 TTS/字幕框。

评委结果页要求：

- 重新设计，不沿用当前简单结果卡片。
- 突出官方获胜方和最佳辩手。
- 用图形展示立论、过程、结辩票。
- 加入适度庆祝动画，但不遮挡核心信息。

学生投票结果页要求：

- 明确标题为“学生投票结果”。
- 突出学生票优胜方。
- 使用柱状图或比例图展示正反方得票。
- 展示最佳辩手排行。

视觉原则：

- 科技舞台 + 书卷感。
- 题目和环节标题用更飘逸字体。
- 信息和计时用高可读无衬线。
- 所有文字必须适配投影环境。

改造任务：

- `P1`：更新 `ScreenScene` 类型。
- `P1`：删除 `TeamsScene` 代码路径。
- `P1`：新增 `PausedScene` 或 live overlay。
- `P1`：拆分 `JudgeCommentaryScene`、`JudgeResultScene`、`AudienceResultScene`。
- `P1`：生成或制作赛事字标资源，保留 CSS fallback。
- `P2`：加入结果页动画，并遵守 `prefers-reduced-motion`。

验收标准：

- 1366x768、1440x900、1920x1080 三种尺寸无溢出。
- 大屏无技术状态文字。
- 评委结果和学生结果不会混淆。

### 4.4 辩手端 Console

现状问题：

- `ConsolePage` 已有身份选择和硬件测试，但人类/AI 辩手主页面仍共用同一套布局。
- 人类发言只有开始/结束，没有暂停/继续。
- AI 辩手页仍可以看到一些不必要的人类发言控制心智。

目标状态：

- `/console` 先进入身份选择。
- 身份选择固定展示 8 个后台预设席位。
- 第二步硬件测试按身份类型变化。
- 人类辩手页和 AI 辩手页分开渲染。

人类辩手页：

- 当前比赛状态。
- 当前阶段。
- 本环节发言对象。
- 是否轮到自己。
- 倒计时。
- 开始发言。
- 暂停发言。
- 继续发言。
- 结束发言。
- 当前转写只在本人发言时显示。

AI 辩手页：

- 当前比赛状态。
- 当前阶段。
- 是否轮到该 AI 席位。
- Agent API 连通状态。
- 现场声音输出端状态说明。
- 轮到该 AI 席位且当前没有其他发言时，在本端启动 Agent 发言。
- 当前版本不提供“让 AI 队友发言”的人类辩手入口；自由辩论只按赛制授权当前方席位，AI 席位由其自己的辩手端启动。
- 不显示麦克风、录音归档、ASR 状态。

Agent 辩手第二步硬件测试只要求检测 API 请求是否连通；扬声器/TTS/铃声不在辩手电脑外放，统一由物理连接大屏/音响的 Host 或 Admin 控制电脑输出。

改造任务：

- `P1`：拆出 `HumanConsoleView` 与 `AgentConsoleView`。
- `P1`：新增发言暂停/继续按钮和接口调用。
- `P1`：新增 AI 辩手端授权启动接口；Host 只展示授权结果，不替 AI 席位启动发言。
- `P1`：隐藏无意义技术状态。
- `P1`：首屏聚焦“现在是否轮到你”。
- `P2`：移动端做纵向兼容。

验收标准：

- 人类辩手开始发言后可暂停、继续、结束。
- AI 辩手只有在赛制授权且当前无其他发言时，才显示可启动 Agent 发言的主按钮。
- 人类辩手页不出现“让 AI 队友发言”，旧 `request-ai-teammate` 能力在 MVP 中显式暂缓。
- AI 辩手页不会出现麦克风测试后的 ASR/录音状态。
- 页面文案全中文，不出现 `expired` 等英文状态。

### 4.5 学生投票 Vote

现状问题：

- `VotePage` 使用计算所 logo。
- 路径仍以 `/vote/:matchId` 为主要心智。
- 投票后前端显示已提交，但后端防重复仍需增强。

目标状态：

- 默认入口 `/vote`。
- 使用人机辩论赛标识。
- 每人只能投一次，投后不可修改。
- 不实时展示投票结果。

改造任务：

- `P1`：改用赛事标识。
- `P1`：默认 `/vote` 调当前比赛。
- `P1`：后端同时保存 token hash 与 IP/UA hash。
- `P1`：重复提交返回友好中文提示。
- `P2`：投票成功页展示简洁感谢，不暴露当前票数变化。

验收标准：

- 同一浏览器重复提交被拒绝。
- 清 localStorage 后，同 IP/UA 重复提交也被拒绝。
- 投票关闭时不能提交。

### 4.6 技术后台 Admin

现状问题：

- `AdminPage` 已有总览、设置、语音、投票、应急、安全、导出模块。
- 但“辩手管理”和“Agent 管理”仍混在比赛设置和 Agent 状态里。
- Agent 仍主要挂在 speaker 字段上，不是独立对象。
- 数据管理尚不足以支撑历史分析。

目标模块：

| 模块 | 主要能力 |
| --- | --- |
| 总览 | 当前状态、流程、设备、Agent、语音、投票、数据健康 |
| 比赛管理 | 当前比赛实例、归档导出、归档并创建新比赛、历史归档入口 |
| 展示与赛制 | 名称、辩题、双方立场、队伍展示、流程规则 |
| 辩手管理 | 固定 8 席位，配置人类/AI、姓名、绑定已有 Agent 配置 |
| Agent 管理 | 新增/编辑/删除 REST Agent 与 OpenAI SDK Agent 配置库 |
| TTS/ASR 管理 | 讯飞配置、诊断、自检、降级、现场声音输出端 |
| 数据管理 | 当前/历史归档、结构化数据概览、导出 |
| 应急干预 | 紧急停止、强制结束、跳过、回滚、AI 中断、转写修订 |
| 访问安全 | token 开关、轮换、入口链接 |

改造任务：

- `P1`：把 Host 不应承担的危险和调试功能全部移到 Admin。
- `P1`：拆出比赛实例管理，避免把“一次比赛生命周期”和展示字段设置混在一起。
- `P1`：拆出固定 8 席位管理。
- `P1`：新增 Agent 配置管理；辩手管理只能绑定已有 Agent，不能在席位中临时新增模型或接口。
- `P1`：新增现场声音输出设置，可指定物理连接大屏/音响的 Host 或 Admin 电脑作为唯一输出端并测试本机铃声。
- `P2`：新增数据管理概览和导出记录。
- `P2`：优化总览状态，覆盖所有现场关键链路。

验收标准：

- 管理员能在后台完成赛前配置。
- Host 不再需要进入 Admin 才能完成正常现场流程。
- Agent 作为独立配置对象管理；AI 席位只保存 `agent_config_id` 绑定，不直接编辑模型、endpoint 或 key。

## 10. 风险与取舍

### 10.1 半自动推进优先于全自动

全自动推进容易在现场误切环节。当前采用半自动：系统到时后结束发言并提示，Host 确认下一步。

### 10.2 历史先导出，不做历史浏览界面

历史浏览界面会扩大范围。本轮先保证完整归档和导出，后续再做历史列表和详情。

### 10.3 投票无感限制不是强实名

localStorage token + IP/UA hash 能减少重复投票，但不能防止换设备、换浏览器、换网络。若以后需要强约束，应引入校内身份或一次性入场码。

### 10.4 固定 8 席位优先

当前比赛是 4v4 人机辩论，固定 8 席位能显著降低 UI 和状态机复杂度。任意人数赛制延后。

### 10.5 OpenAI SDK Agent key 不进入前端

SDK Agent 的 API key 只允许通过后端环境变量引用，不在后台页面展示明文，不进入 WebSocket 快照，不进入导出包。

### 10.6 大屏动效必须服务信息

结果页可以有仪式感，但现场投影的第一优先级是清楚。动效不得遮挡优胜方、票数、最佳辩手等核心信息。

### 10.7 自由辩论 AI 队友请求暂缓

早期原型里的“让 AI 队友发言”会让人类辩手端承担调度职责，容易和“Host 只确认流程、辩手端只操作本人席位”的现场边界冲突。本轮只允许被赛制自动授权的 AI 席位在自己的辩手端启动 Agent 发言；队友代触发、AI 自动接管和投机预生成全部延后。
