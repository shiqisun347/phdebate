# Admin 后台重构需求规格

> 本文是 admin 后台**完全重构**的需求规格，不在现有 `AdminPage.tsx` 上增量修改。
> 现有页面仅作为后端接口与业务逻辑的参考，不作为 UI/交互参考。

---

## 0. 重构原则

1. **完全重构**：不基于当前 admin 修改，参考成熟优秀的后台管理系统，提供更好的用户体验、界面设计，同时保证功能完整、易用。
2. **基础框架**：基于 `shadcn-admin`（satnaing/shadcn-admin，技术栈 React + Vite + TypeScript + Tailwind CSS + shadcn/ui）改进、设计与开发。该框架与现有前端工程（Vite/React/TS）一致，并自带主题/皮肤切换能力，便于落地下文的"风格切换"需求。
   - 备选重型方案：Ant Design Pro（如需更完整的企业级表格/表单体系再评估）。
3. **不要把功能铺开展示**：默认收敛、按需展开；用清晰的分区、导航与渐进式表单（多步骤向导）替代"一屏堆满"的布局。
4. **风格切换**：用户可在 2~3 种色彩风格中选择，满足不同审美；切换后全局生效并持久化。
5. **剔除不需要的功能**：废弃独立的 `host` 页面（见 §3.3），将其控场职责并入 admin 子页。

---

## 1. 整体信息架构（三大分区）

进入 admin 后，用户必须**一眼看清三大分区**及其职责边界。三大分区通过主导航（侧边栏分组）明确区隔：

| 分区 | 定位 | 与具体比赛的关系 |
| --- | --- | --- |
| **A. 全局设置** | 面向整个系统的通用管理 | 与具体比赛无关 |
| **B. 当前比赛管理** | 配置"当前激活比赛"的辩手与流程 | 绑定当前比赛 |
| **C. 现场控制** | 赛前调试 + 现场控场 | 绑定当前比赛、面向现场执行 |

---

## 2. A · 全局设置

针对整个系统、与具体比赛无关的通用管理。包含以下模块：

### 2.1 比赛管理
- 比赛的增、删、改、查。
- **新增比赛 = 多阶段设置向导（Wizard）**，分步骤完成：
  1. **基础信息**：用于基础信息与大屏（screen）展示。
     - 比赛名称（screen 左上角，支持**图片或文字**两种方式）
     - 机构信息（screen 右上角，支持**图片或文字**两种方式）
     - 辩题
  2. **赛制规则**：只能从预设赛制规则中**选择**，不能自定义（自定义在 §2.2 赛制规则管理中完成）。
  3. **完成**：创建后询问"是否切换为当前比赛"。
- 列表需展示各比赛状态，并支持一键"切换为当前比赛"。

### 2.2 赛制规则管理
管理员可增删改查赛制规则。每条规则包含：
- 赛制名称
- 赛制简介
- **赛制流程**：提供一份**可复制的流程模板**，用户基于模板修改文本 → 后台调用 AI 生成结构化规则 → 向用户展示**程序流程图**（可视化流程）。
- 其他必要信息

### 2.3 Agent 管理
管理员可增删改查 agent。每个 agent 包含：
- agent 名称
- agent 类型：`REST API agent` / `SDK agent`
- 请求方式：GET / POST 等
- 接口地址（REST API agent 时）
- 模型信息（SDK agent 时，如模型名称等）
- 其他必要信息
- **输入/输出交互 JSON**：用文本框提供可编辑的 JSON 格式（请求体/响应体模板），见 §5 参考格式。

### 2.4 TTS / ASR 管理
- 设置与调试 TTS / ASR 相关参数（API 凭据、endpoint、语种、音色等）。
- **现场测试**：
  - TTS：输入一段文本 + 选择音色 → 合成 → 用**扬声器**播放，验证效果。
  - ASR：使用**麦克风**说一段话 → 系统识别 → 展示识别文本，验证效果。
- 用文本框提供输入/输出交互的 JSON 格式。
- 测试用讯飞凭据见 §6。

### 2.5 数据管理
- 查看、管理、下载历史所有比赛的相关数据：比赛记录、辩手信息、投票结果等。
- 清晰的数据展示界面，便于查看与分析。
- 支持必要的数据操作：导出数据、删除数据等。

### 2.6 安全管理
- 开启"安全登录"时：管理员设置密码，用户须凭密码才能进入**后台管理页面**和**辩手页面**。
- 未开启时：可直接进入。
- 该密码对**所有比赛**通用。

### 2.7 小七管理（自研智能体）
- 设置小七的相关信息与功能 prompt：自我介绍、评价辩论、给出结果、自定义问题。
- 每个功能可**直接点击测试**，查看小七表现。
- **系统只负责发送命令/请求**；小七发音依赖小七自身。请求体、请求地址在此配置。
- 设置小七的**形象图**（后续由用户提供，用于小七观点页/结果页）。

---

## 3. B · 当前比赛管理

针对"当前激活比赛"的配置。

### 3.1 辩手管理
- 根据当前赛制规则确定辩手数量，在对应位置设置辩手（与现有辩手管理逻辑相近即可）。

### 3.2 比赛流程管理
- 根据当前赛制规则设置比赛流程。流程为**预设**，用户只能选择、不能自定义。
- 每个阶段的**发言时间**与**发言人**也是预设，用户只能选择、不能自定义。
- 提供**可视化界面**清晰展示每个阶段的发言时间与发言人，并便于查看/调整（在预设范围内）。

### 3.3 控场页面（废弃 host）
- **不再保留独立 host 页面**：使用 admin 的一个子页作为控场页面，由后台管理人员兼任控场人员。

---

## 4. C · 现场控制

### 4.1 调试与总览页面
- 提供调试与总览页面，检测所有**设备、功能**的状态。
- **仅用于赛前调试/测试**，与"比赛过程检测"无关，确保比赛中不出现设备或功能问题。

### 4.2 控场页面（现场流程编排）

部署形态：
- 一台电脑进入 **screen 页面**作为主屏幕（展示比赛信息/内容、输出声音）。
- 一台电脑进入 **admin 页面**作为控场页面。

**控场页面包含的模块**：
- **AI 辩手控制模块**：罗列当前所有 AI 辩手小块，点击每个 AI 辩手提供以下功能：
  - 自我介绍
  - 替代方案（紧急情况下，使用替代方案进行**单次回答**）
  - 重试（仅在轮到该 agent 发言时有效，点击后重新向 agent 发送发言请求；否则右下角提示"并非该 agent 回答环节"）
  - 其他必要功能（精简，不冗余）
- **小七控制模块**：小七自我介绍、评价辩论、给出结果、自定义问题。
- **实时辩论过程模块**：类似聊天框，展示全局唯一辩论过程。优先级低于页面控制，可下拉查看，不要求出现在首屏。

**现场流程（控场人员操作时序）**：
1. 控场人员先进入**调试与总览页面**完成设备/功能调试。
2. 进入控场页面。主持人介绍辩手；介绍到 AI 辩手时，控场人员在 AI 辩手控制模块中选择该 agent 进行**自我介绍**。
3. 介绍完，主持人宣布开始，控场人员点击"开始比赛"→ 主屏幕切换到**实况界面**，按预设流程进行，自动控制发言时间与发言人切换。
4. 控场人员可随时**暂停/继续**比赛，**重置当前发言**或**重置整个比赛**。
5. 比赛结束，主持人介绍小七 → 控场人员切换到**小七观点页面**（设计同评委点评页面，仅将二维码区域换成小七形象图）。
6. 主持人宣布小七开始点评 → 控场人员点击**辩论点评**发送请求（发音依赖小七自身，系统只发请求）。
7. 点评结束，主持人请小七评价获胜方与最佳辩手 → 控场人员点击小七**辩论结果**发送请求 → 控场人员**人工选择**对应队伍与辩手并保存 → 切换到**小七结果页面**（设计参考评委结果页面，突出小七评价结果与获胜方）。
8. 主持人宣布评委点评与学生投票开始 → 控场人员切换到**评委点评界面**，学生开始投票。
9. 主持人手动切换到**评委结果界面**，宣布获胜方，同时**自动结束**学生投票。
10. 主持人手动切换到**学生投票结果界面**，展示学生投票结果与最终获胜方。
